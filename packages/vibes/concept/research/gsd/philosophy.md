# Design Philosophy from GSD

## 1. Solo Developer + Claude Model

> "Planning for ONE person (the user) and ONE implementer (Claude). No teams, stakeholders, ceremonies, coordination overhead. User = visionary/product owner, Claude = builder."

Anti-enterprise patterns GSD explicitly deletes if found:
- Team structures and RACI matrices
- Stakeholder management plans
- Sprint ceremonies
- Human dev time estimates
- Documentation for documentation's sake

**Why it matters:** Most planning systems are designed for teams. GSD is designed for a user-Claude dyad. That's fundamentally different — fewer approval gates, less ceremony, faster iteration.

---

## 2. Complexity Hidden in the System

> "The complexity is in the system, not in your workflow. Behind the scenes: context engineering, XML prompt formatting, subagent orchestration, state management. What you see: a few commands that just work."

The user never sees: wave computation, context budgeting, fresh subagent spawning, checkpoint classification. They just see `/gsd:execute-phase 3`.

**Applicable to vibes:** The vibes agent should feel simple from the outside. "Build a README for this repo." All the context loading, subagent delegation, and verification should be invisible.

---

## 3. Ship Fast Over Process

> "Plan → Execute → Ship → Learn → Repeat"

Only 4 steps before shipping. No approval gates, no documentation requirements, no retrospectives.

The system is designed to minimize time from idea to working artifact. Research is fast. Planning is parallel. Execution uses fresh contexts. Verification is automated.

---

## 4. Training Data as Hypothesis

> "Claude's training is 6–18 months stale. Treat pre-existing knowledge as hypothesis, not fact."

Research agents verify everything against Context7 and official docs before stating facts. LOW confidence findings are flagged, not asserted.

**Applicable to vibes:** When generating technical documentation or tickets, don't rely on Claude's knowledge of library APIs. Verify against current docs.

---

## 5. Plans Are Prompts

> "PLAN.md IS the prompt (not a document that becomes one)."

This reframes planning entirely. The output of a planning session is not a requirements document that someone then translates into a prompt. The plan is the prompt, written in Claude's execution language.

**Applicable to vibes:** Tickets generated by vibes should be Claude-executable. Not "implement a login page" but "create src/app/login/page.tsx, use the auth context from src/contexts/auth.tsx, redirect to /dashboard on success."

---

## 6. Context as First-Class Architecture

Context engineering is not an afterthought — it drives the entire system design.

Every architectural decision traces back to context management:
- Files are small because small files are read, large files are skipped
- Subagents are spawned because fresh contexts perform better than degraded ones
- State is in STATE.md because a single file is always loadable
- Tasks use XML because prose is ambiguous in Claude's parsing
- Plans are short (2–3 tasks) because longer plans degrade mid-execution

**The central claim:** Quality of Claude output is primarily determined by quality of context, not quality of prompting.

---

## 7. Dream Extraction, Not Requirements Gathering

> "Project initialization is dream extraction, not requirements gathering."

The init phase is about understanding what the user is excited about, what problem they saw, what success looks like to them emotionally — not about eliciting a formal requirements list.

Formal requirements emerge from the dream, not the other way around.

**Applicable to vibes:** The opening interaction should feel like a conversation with an interested partner, not a product manager filling out a form.
