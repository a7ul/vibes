This is the feature pydantic-ai is most known for - and Vibes implements the same pattern in TypeScript.

The idea: declare what your agent needs as a type (`TDeps`), inject it at run time, and access it from _everywhere_ - system prompts, tools, validators, toolsets, history processors. One consistent interface, zero globals.

> **Coming from pydantic-ai?** `RunContext<TDeps>` maps directly to pydantic-ai's `RunContext[TDeps]`. The `deps` field is your injected state. Everything that received `RunContext` in pydantic-ai receives it here too.

## The Hierarchy

Here's everywhere `RunContext` flows through in Vibes - from least to most granular:

| Where | How | pydantic-ai equivalent |
|---|---|---|
| Agent construction | `systemPrompt: (ctx) => string` | `system_prompt=` argument |
| Added dynamically | `agent.addSystemPrompt((ctx) => string)` | `@agent.system_prompt` |
| Per-run instructions | `instructions: (ctx) => string` | `instructions=` argument |
| Tool execution | `execute: (ctx, args) => result` | `@agent.tool` |
| Tool visibility | `prepare: (ctx) => tool \| null` | - |
| Dynamic approval | `requiresApproval: (ctx, args) => boolean` | - |
| Result validation | `resultValidators: [(ctx, output) => output]` | `@agent.result_validator` |
| Toolset resolution | `toolset.tools(ctx)` | - |
| History processing | `historyProcessors: [(messages, ctx) => messages]` | - |

All of these receive the same `RunContext<TDeps>`:

```ts
interface RunContext<TDeps> {
  deps: TDeps;         // your injected dependencies
  usage: Usage;        // token usage accumulated so far this run
  runId: string;       // unique ID for this run
  retryCount: number;  // how many validation retries have occurred
  toolName: string | null;  // set inside tool execute()
  metadata: Record<string, unknown>;  // arbitrary run metadata
}
```

Inject deps at run time:

```ts
const result = await agent.run("Do something", {
  deps: {
    db: new Database(process.env.DATABASE_URL!),
    user: currentUser,
    config: appConfig,
  },
});
```

## 1. System Prompts

The most common use - personalise the prompt with runtime state.

### At construction

```ts
type Deps = { user: User; locale: string };

const agent = new Agent<Deps>({
  model,
  systemPrompt: (ctx) =>
    `You are an assistant for ${ctx.deps.user.name}. Respond in ${ctx.deps.locale}.`,
});
```

### `agent.addSystemPrompt()` - the `@agent.system_prompt` equivalent

This is the direct equivalent of pydantic-ai's `@agent.system_prompt` decorator. Call it any time after construction - in middleware, plugins, or to build agents incrementally:

```ts
const agent = new Agent({ model, systemPrompt: "You are a helpful assistant." });

// Static addition
agent.addSystemPrompt("Always be concise.");

// Dynamic - receives RunContext every run
agent.addSystemPrompt((ctx) => `Today is ${new Date().toDateString()}.`);
agent.addSystemPrompt((ctx) => `Current user: ${ctx.deps.user.name}`);
agent.addSystemPrompt(async (ctx) => {
  const plan = await ctx.deps.db.plans.getCurrent(ctx.deps.user.id);
  return `User is on the ${plan.name} plan.`;
});
```

> **pydantic-ai mapping:**
> ```python
> @agent.system_prompt
> async def inject_plan(ctx: RunContext[Deps]) -> str:
>     plan = await ctx.deps.db.plans.get_current(ctx.deps.user.id)
>     return f"User is on the {plan.name} plan."
> ```

All system prompts - static and dynamic - are joined with `\n\n` into a single system message each run.

### `instructions`

A second slot for dynamic context. Works identically to `systemPrompt` but appended after it. Use `systemPrompt` for the stable base and `instructions` for per-run state:

```ts
const agent = new Agent<Deps>({
  model,
  systemPrompt: "You are a support assistant.",          // stable
  instructions: (ctx) => `Ticket #${ctx.deps.ticketId}`, // per-run
});
```

## 2. Tools

Every tool's `execute` function receives `RunContext` as its first argument. This is the equivalent of pydantic-ai's `@agent.tool` decorator.

```ts
type Deps = { db: Database; emailService: EmailService };

const lookupUser = tool<Deps>({
  name: "lookup_user",
  description: "Find a user by email",
  parameters: z.object({ email: z.string().email() }),
  execute: async (ctx, { email }) => {
    // ctx.deps is fully typed
    const user = await ctx.deps.db.users.findByEmail(email);
    if (!user) throw new Error(`No user found for ${email}`);
    return user;
  },
});
```

> **pydantic-ai mapping:**
> ```python
> @agent.tool
> async def lookup_user(ctx: RunContext[Deps], email: str) -> User:
>     return await ctx.deps.db.users.find_by_email(email)
> ```

### `prepare` - dynamic tool visibility

Called before every model turn. Return `null` to hide the tool based on context, or a modified definition to change its description or parameters:

```ts
const sensitiveSearch = tool<Deps>({
  name: "search_pii",
  description: "Search records including PII",
  parameters: z.object({ query: z.string() }),
  prepare: async (ctx) => {
    const allowed = await ctx.deps.db.permissions.canAccessPII(ctx.deps.user.id);
    return allowed ? undefined : null; // null = hide tool this turn
  },
  execute: async (ctx, { query }) => ctx.deps.db.searchPII(query),
});
```

### `requiresApproval` - dynamic human-in-the-loop

Use a function to decide per-call whether approval is needed:

```ts
const deleteRecord = tool<Deps>({
  name: "delete_record",
  description: "Delete a database record",
  parameters: z.object({ id: z.string(), environment: z.string() }),
  requiresApproval: (ctx, args) =>
    args.environment === "production" || ctx.deps.user.role !== "admin",
  execute: async (ctx, { id }) => ctx.deps.db.delete(id),
});
```

## 3. Result Validators

Result validators are the equivalent of pydantic-ai's `@agent.result_validator`. They receive `RunContext` and the parsed output, and can validate, transform, or throw to trigger a retry.

```ts
type Deps = { db: Database };

const agent = new Agent<Deps, { itemId: string }>({
  model,
  outputSchema: z.object({ itemId: z.string() }),
  resultValidators: [
    async (ctx, output) => {
      // Validate the output references a real database record
      const exists = await ctx.deps.db.items.exists(output.itemId);
      if (!exists) throw new Error(`Item ${output.itemId} not found - try again`);
      return output; // return (optionally transformed) output
    },
  ],
});
```

> **pydantic-ai mapping:**
> ```python
> @agent.result_validator
> async def validate_item(ctx: RunContext[Deps], output: Output) -> Output:
>     if not await ctx.deps.db.items.exists(output.item_id):
>         raise ModelRetry(f"Item {output.item_id} not found")
>     return output
> ```

Multiple validators run in order. Any validator can throw to trigger a retry (up to `maxRetries`). Add validators after construction with `agent.addResultValidator()`.

## 4. Toolsets

Toolsets resolve their tool list on every turn via `tools(ctx)`. This is the primary mechanism for context-aware tool selection:

```ts
type Deps = { user: { role: "admin" | "viewer" }; db: Database };

class UserToolset implements Toolset<Deps> {
  async tools(ctx: RunContext<Deps>) {
    const perms = await ctx.deps.db.permissions.get(ctx.deps.user.id);

    const base = [readTool, searchTool];
    if (perms.canWrite) base.push(writeTool);
    if (perms.canDelete) base.push(deleteTool);
    return base;
  }
}
```

`FilteredToolset` wraps any toolset with a predicate:

```ts
const adminTools = new FilteredToolset(
  new FunctionToolset([nukeEverythingTool]),
  (ctx) => ctx.deps.user.role === "admin",
);
```

## 5. History Processors

History processors transform the message history before each model turn. They receive `RunContext` so they can make context-aware decisions:

```ts
import { privacyFilterProcessor, trimHistoryProcessor } from "@vibes/framework";

const agent = new Agent<Deps>({
  model,
  historyProcessors: [
    trimHistoryProcessor(20),         // keep last 20 messages
    privacyFilterProcessor([          // redact sensitive patterns
      { type: "regex", pattern: /\b\d{16}\b/, redactValue: "[CARD]" },
    ]),
  ],
});
```

For fully dynamic processors:

```ts
agent.addHistoryProcessor(async (messages, ctx) => {
  // e.g. load summaries from the database for this user
  const summary = await ctx.deps.db.summaries.get(ctx.deps.user.id);
  return [{ role: "system", content: summary }, ...messages.slice(-10)];
});
```

## Putting It All Together

A realistic agent that uses RunContext throughout:

```ts
type Deps = {
  db: Database;
  user: User;
  permissions: PermissionService;
};

const agent = new Agent<Deps>({
  model: anthropic("claude-sonnet-4-6"),

  // Base system prompt
  systemPrompt: "You are a helpful assistant.",

  // Dynamic context injected per run
  instructions: (ctx) =>
    `You are helping ${ctx.deps.user.name} (${ctx.deps.user.role}).`,

  // Tools that use deps
  tools: [lookupTool, writeTool],

  // Toolset that filters based on permissions
  toolsets: [
    new FilteredToolset(adminToolset, async (ctx) =>
      ctx.deps.permissions.isAdmin(ctx.deps.user.id)
    ),
  ],

  // Validate output against the database
  resultValidators: [
    async (ctx, output) => {
      const valid = await ctx.deps.db.validate(output);
      if (!valid) throw new Error("Output references unknown records");
      return output;
    },
  ],
});

// Add more system prompts dynamically (e.g. in middleware)
agent.addSystemPrompt(async (ctx) => {
  const plan = await ctx.deps.db.plans.getCurrent(ctx.deps.user.id);
  return `User plan: ${plan.name}. Remaining credits: ${plan.credits}.`;
});

// Inject everything at run time
const result = await agent.run("Help me with my account", {
  deps: {
    db: database,
    user: currentUser,
    permissions: permService,
  },
});
```

## Testing

DI makes testing straightforward - swap real deps with fakes:

```ts
const fakeDeps: Deps = {
  db: {
    users: { findByEmail: async (email) => ({ id: "u1", name: "Alice", email }) },
    plans: { getCurrent: async () => ({ name: "Pro", credits: 100 }) },
    validate: async () => true,
  },
  user: { id: "u1", name: "Alice", role: "admin" },
  permissions: { isAdmin: async () => true },
};

const result = await agent
  .override({ model: new TestModel() })
  .run("test", { deps: fakeDeps });
```
