Thank you for your interest in contributing to Vibes. This guide covers everything you need to get from zero to a merged pull request.

## Prerequisites

Vibes is a Deno-native project. You need:

- **[Deno](https://deno.land/)** (latest stable) â€” the primary runtime; Node.js is not required
- **Git**

<Note>
  Node.js is not required. Vibes uses Deno for all scripts, tests, and tooling.
</Note>

## Repository Setup

```bash
git clone https://github.com/a7ul/vibes
cd vibes
```

The repository is organized as a monorepo:

| Path | Contents |
|------|----------|
| `packages/framework/` | Framework source code (`lib/`) and documentation (`docs/`) |
| `packages/framework/lib/` | Core framework implementation |
| `packages/framework/docs/` | Mintlify MDX documentation |

## Running Tests

Tests live in the `packages/framework/` directory and are written for Deno.

```bash
cd packages/framework
deno test
```

To run with permissions explicitly:

```bash
deno test --allow-env --allow-net
```

<Info>
  Integration tests that call real LLM providers require API keys. Set `ANTHROPIC_API_KEY`, `OPENAI_API_KEY`, or the relevant provider key as environment variables before running those tests. Unit tests and tests using `TestModel` or `FunctionModel` run without any API keys.
</Info>

## Making Changes

### Branch Naming

Use descriptive branch names that communicate the kind of change:

| Prefix | When to use |
|--------|-------------|
| `feat/` | New feature or API (`feat/add-mcp-streaming`) |
| `fix/` | Bug fix (`fix/runcont-type-inference`) |
| `docs/` | Documentation only (`docs/update-tools-page`) |
| `refactor/` | Code restructuring without behavior change |
| `test/` | New or updated tests |
| `chore/` | Tooling, deps, config (`chore/update-deno-lock`) |

### Commit Messages

Follow [Conventional Commits](https://www.conventionalcommits.org/):

```
feat: add streaming support for structured output
fix: resolve null ref in RunContext when deps is undefined
docs: add Mermaid diagram to graph.mdx
```

Types: `feat`, `fix`, `docs`, `refactor`, `test`, `chore`, `perf`, `ci`

### Documentation for API Changes

Every public API change requires a documentation update in the same PR:

- New exports â†’ add to the relevant concept page under `docs/concepts/`
- New integration â†’ add a page under `docs/integrations/`
- Bug fixes that change observable behavior â†’ update the relevant page with a note

## Submitting a Pull Request

1. Fork the repository on GitHub
2. Create a branch from `main`:
   ```bash
   git checkout -b feat/your-feature-name
   ```
3. Make your changes, commit them following the convention above
4. Push to your fork:
   ```bash
   git push origin feat/your-feature-name
   ```
5. Open a pull request against `main` on [github.com/a7ul/vibes](https://github.com/a7ul/vibes)

In your PR description:
- Explain **what** changed and **why**
- If adding a new API, include a minimal usage example
- If fixing a bug, describe how to reproduce the original issue
- Include documentation changes in the same PR as the code changes

## Documentation Contributions

Documentation lives in `packages/framework/docs/` and is built with [Mintlify](https://mintlify.com/).

### Preview Docs Locally

```bash
cd packages/framework/docs
mintlify dev
```

This starts a local dev server at `http://localhost:3000`.

### Documentation Standards

- **All new concept pages need at least one Mermaid diagram** â€” architecture flows, sequence diagrams, and decision trees all help readers understand quickly
- **API names must be verified against the source** â€” check `packages/framework/lib/` directly; do not copy from third-party reference pages or old docs that may be stale
- **MDX format** â€” pages use standard Mintlify frontmatter (`title`, `description`) and Mintlify components (`<Info>`, `<Note>`, `<Warning>`, `<Card>`, `<CardGroup>`)
- **Code examples must be runnable** â€” if an example uses a real provider, use the Open-Meteo weather API (free, no API key) or a similar zero-cost service for demos

### Adding a New Page

1. Create the MDX file in the appropriate subdirectory under `docs/`
2. Add it to `docs/docs.json` in the correct navigation group
3. Verify the page renders correctly with `mintlify dev`

## Code of Conduct

Vibes follows standard open-source community norms:

- Be welcoming and respectful
- Focus on the work, not the person
- Assume good faith

If you experience or witness behavior that violates these norms, open an issue on [GitHub](https://github.com/a7ul/vibes/issues).

## Reporting Bugs

Open an issue at [github.com/a7ul/vibes/issues](https://github.com/a7ul/vibes/issues). Include:

- Vibes version (`@vibes/framework` version from your lock file)
- Deno version (`deno --version`)
- Minimal reproduction case
- Expected vs. actual behavior
