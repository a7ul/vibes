## 1. Language & Runtime

- **TypeScript** (strict mode) running on **Deno**
- No `tsconfig.json` ‚Äî Deno's built-in TS handling is used
- Imports use `.ts` extensions explicitly (Deno requirement)
- NPM packages imported via `npm:` specifier in `deno.json` import map

## 2. Naming Conventions

| Element | Convention | Example |
|---------|-----------|---------|
| Files | `snake_case.ts` | `mcp_toolset.ts`, `_run_utils.ts` |
| Internal files | `_` prefix | `_run_utils.ts`, `_helpers.ts` |
| Classes | `PascalCase` | `Agent`, `Graph`, `DeferredToolRequests` |
| Interfaces | `PascalCase` | `Toolset`, `ToolDefinition`, `RunContext` |
| Type aliases | `PascalCase` | `HistoryProcessor`, `EndStrategy` |
| Functions | `camelCase` | `executeRun()`, `createRunContext()` |
| Constants | `camelCase` or `UPPER_SNAKE` (for true constants) | |
| Test files | `{module}_test.ts` | `agent_test.ts`, `mcp_toolset_test.ts` |
| Barrel files | Always `mod.ts` | `lib/toolsets/mod.ts` |

## 3. Import Style

- **Types**: Use `import type { ... }` for type-only imports (enforced by Deno lint)
- **Values**: Standard `import { ... }` for runtime values
- **Barrels**: Use `mod.ts` within submodules; external consumers import from `mod.ts`
- **No default exports** ‚Äî all exports are named

```typescript
// CORRECT
import type { RunContext } from "../types/context.ts";
import { executeRun } from "./execution/run.ts";

// WRONG
import RunContext from "../types/context.ts";  // no defaults
```

## 4. Type Patterns

### Generic Parameters
- `TDeps` ‚Äî dependency injection type for `RunContext`
- `TOutput` ‚Äî output type for `Agent<TDeps, TOutput>` and `RunResult<TOutput>`
- Generic defaults to `undefined` for deps, `string` for output

### Zod Integration
- Tool parameters always typed as `ZodType` (not raw JSON Schema)
- `z.infer<ZodType>` used to derive TypeScript types from Zod schemas
- Output schemas validated with Zod; validation errors trigger model nudge+retry

### Discriminated Unions
- Events use `kind` discriminant: `{ kind: "node" | "output" }`
- Graph steps: `GraphStep<TState, TOutput>` with `kind: "node" | "output"`

---

## 5. Immutability Patterns

- **Agent**: `agent.clone()` and `agent.override()` return new `Agent` instances; existing agent is never mutated
- **HistoryProcessors**: Must return new message arrays, never mutate in-place
- **RunContext**: Created fresh per execution; not shared across runs
- Tool `execute` functions should not mutate their `args` parameter

## 6. Error Handling

### Custom Error Types (`lib/types/errors.ts`)
- `ApprovalRequiredError` ‚Äî thrown when deferred tools need human approval; carries `DeferredToolRequests`
- `MaxTurnsError` ‚Äî thrown when max turns exceeded
- `MaxGraphIterationsError` ‚Äî thrown by graph FSM
- `UnknownNodeError` ‚Äî unknown node ID in graph

### Pattern
```typescript
// Catch specific errors at the call site
try {
  const result = await agent.run(prompt);
} catch (err) {
  if (err instanceof ApprovalRequiredError) {
    const deferred = err.deferred;
    // inspect deferred.requests, then call agent.resume()
  }
  throw err;  // re-throw unknown errors
}
```

### Tool Errors
- Tools can `throw` to signal failure
- `maxRetries` on `ToolDefinition` controls automatic retry count
- `argsValidator` throws to reject args without consuming a retry

## 7. Module Organization

- **Feature/domain grouping**: `mcp/`, `graph/`, `temporal/`, `otel/`, `multimodal/` ‚Äî each owns its types, logic, and exports
- **Types separated**: Pure types live in `lib/types/` ‚Äî no runtime dependencies, safe to import anywhere
- **Testing utilities** in `lib/testing/` ‚Äî exported for consumer use (not just internal)
- Each feature module has a `mod.ts` barrel; `lib/` root does NOT have a barrel (only `packages/framework/mod.ts`)

---

## 8. Async Patterns

- All async operations use `async/await` (no raw Promise chains)
- Streaming uses async generators / `AsyncIterable`
- Tool `execute` functions are always `async` even if synchronous internally

---

## 9. Documentation Style

- **JSDoc on public API**: Classes, interfaces, and exported functions have JSDoc comments
- **Inline comments** for non-obvious logic (especially in `_run_utils.ts`)
- **`@example` blocks** in JSDoc for key types (`TestModel`, `HistoryProcessor`, etc.)
- No excessive comments on obvious code

---

## 10. Configuration Pattern

- `AgentOptions` is a plain interface (not a class) ‚Äî constructed with object literal
- Optional fields use `?` not union with `undefined`
- Defaults resolved at call time in execution utilities (not stored on Agent)
