# CONCERNS.md — Technical Debt & Issues

_Focus: concerns | Generated by gsd-codebase-mapper_

---

## 1. Technical Debt

### Mutation-Prone APIs
- `Agent` and `FunctionToolset` APIs allow mutable state patterns that could cause hidden side effects
- Tool registration mutates internal state rather than returning new instances

### Unsafe Type Assertions
- MCP client code uses `as` type assertions that bypass TypeScript's type safety
- Content handling uses type coercion that could silently fail at runtime

### Large Files
- `packages/framework/lib/execution/_run_utils.ts` — 828 lines, violates file size conventions
  - Should be split into focused modules: streaming, history, tool dispatch
- `packages/framework/lib/agent.ts` — complex orchestration logic mixed with configuration

---

## 2. Fragile Areas

### Deferred Tool Approval Flow
- The deferred tool approval + resume mechanism is highly complex
- Edge cases in approval state persistence could cause tools to be silently skipped
- No clear error path when approval times out or is denied mid-execution

### History Processor Edge Cases
- History processing for multi-turn conversations has subtle edge cases
- Resume/continuation flows may lose context when switching between deferred and direct tools

### MCP Error Handling
- MCP client errors are swallowed in several places (`mcp_toolset.ts`)
- Silent failures mean broken MCP servers may appear to work but return no tools
- No fallback or retry logic for transient MCP connection failures

---

## 3. Performance Concerns

### Token Counting Without Caching
- Token counting appears to be done on every request without memoization
- Large system prompts recounted repeatedly across turns

### Unbounded Queues
- Semaphore implementation for concurrent tool execution has unbounded queue
- Could cause memory pressure under high load or long-running agents

### Streaming Buffer Issues
- Streaming output buffering may accumulate large amounts of data in memory
- No explicit backpressure mechanism for slow consumers

---

## 4. Security Risks

### External Tool Bypass
- `external_toolset.ts` and `wrapper_toolset.ts` wrap external tools with minimal validation
- Could allow external tools to bypass permission checks if wrappers are misconfigured

### Dangerous Permission Mode
- `allowDangerouslySkipPermissions: true` is used in v2 autonomous runs
- This is intentional but should be clearly documented as a security boundary

### MCP Server Trust
- MCP servers are treated as trusted by default; no validation of tool schemas from untrusted servers

---

## 5. Test Coverage Gaps

### Under-Tested Flows
- Deferred approval + resume flows lack dedicated tests (`binary_content_test.ts`, `temporal_test.ts` focus on other areas)
- History processor edge cases (empty history, truncated history) not explicitly tested
- MCP error paths: disconnection, malformed responses, timeout scenarios

### Missing Integration Tests
- No end-to-end tests for multi-agent orchestration (sub-agent spawning and result collection)
- No tests for streaming output correctness under concurrent tool execution

### Test Model Limitations
- `test_model.ts` mock may not accurately simulate all real SDK behaviors
- Tests passing against mock may diverge from real API behavior

---

## 6. Known Issues

### v1 vs v2 Divergence
- The Python v1 (`src/vibes/`) and TypeScript v2 (`v2/`) are separate implementations with no shared logic
- Feature parity is claimed (recent commit) but not verified by cross-version tests
- Maintenance burden of two codebases will grow over time

### Output Schema Handling
- `output_schema.ts` output validation may be brittle for complex nested schemas
- No clear strategy for schema evolution or backwards compatibility
