## 1. Architecture Pattern

**Library / Framework** вҖ” `@vibes/framework` is a TypeScript framework for building LLM agents on top of the [Vercel AI SDK](https://sdk.vercel.ai/). It is NOT an application; it is a composable toolkit consumed by downstream agents and apps.

The design philosophy:
- **Model-agnostic**: Uses `LanguageModel` from Vercel AI SDK вҖ” works with Anthropic, OpenAI, etc.
- **Composable toolsets**: Tools are grouped into `Toolset` interfaces, combined at run-time
- **Immutable history**: Message history is never mutated in-place; processors transform copies
- **Type-safe outputs**: Zod schemas validate structured model outputs

## 4. Data Flow (Normal Run)

```
User calls agent.run(prompt, opts)
  вҶ“

  вҶ“
createRunContext() вҶ’ RunContext<TDeps>  (carries deps, usage, runId)
  вҶ“
Loop (up to maxTurns):
  prepareTurn() вҶ’ resolves tools from toolsets + HistoryProcessors applied
  вҶ“
  generateText() [Vercel AI SDK] вҶ’ model response
  вҶ“
  Tool calls dispatched (concurrent unless sequential=true)
  вҶ“
  If tool.requiresApproval вҶ’ throw ApprovalRequiredError (DeferredToolRequests)
  If tool.isOutput вҶ’ collect final result
  вҶ“
  Accumulate messages, update usage
  вҶ“
runValidators() вҶ’ ResultValidator applied
  вҶ“
Return RunResult<TOutput>
```

## 5. Key Abstractions

### Agent (`lib/agent.ts`)
- Central configuration object (model, systemPrompt, tools, toolsets, maxTurns, etc.)
- Immutable: `agent.clone()` and `agent.override()` return new instances
- Does NOT hold mutable state вҖ” state lives in `RunContext` per execution

### ToolDefinition (`lib/tool.ts`)
- Core unit of capability: `{ name, description, parameters: ZodType, execute }`
- Optional: `prepare()` (per-turn filtering), `sequential`, `requiresApproval`, `isOutput`
- Binary content support: tools can return `BinaryContent` or `UploadedFile`

### Toolset (`lib/toolsets/toolset.ts`)
- Interface: `tools(ctx) вҶ’ ToolDefinition[]` вҖ” resolved each turn
- Implementations: `FunctionToolset`, `CombinedToolset`, `FilteredToolset`, `PrefixedToolset`, `PreparedToolset`, `ApprovalRequiredToolset`, `ExternalToolset`, `WrapperToolset`, `MCPToolset`

### HistoryProcessor (`lib/history/processor.ts`)
- Function: `(messages, ctx) вҶ’ messages` вҖ” transforms history before each model call
- Built-ins: `trimHistoryProcessor(n)`, `tokenTrimHistoryProcessor(maxTokens)`, `summarizeHistoryProcessor`
- Applied in order; do NOT mutate вҖ” return new arrays

### RunContext (`lib/types/context.ts`)
- Per-run context carrying `deps`, `usage`, `runId`, `metadata`
- Passed to every tool execute and history processor

### Graph (`lib/graph/`)
- FSM: nodes are `BaseNode` instances implementing `run(state) вҶ’ {next, output}`
- `Graph.runIter()` returns async iterator of steps
- Optional `StatePersistence` for resumable graph runs

### DeferredToolRequests (`lib/execution/deferred.ts`)
- Returned via `ApprovalRequiredError` when tools need human approval
- Caller inspects `.requests`, then calls `agent.resume(deferred, results)`
- `ResumeState` is opaque вҖ” callers pass it back unchanged

## 6. Concurrency Model

- Tools run concurrently by default (all tool calls in one model turn execute in parallel)
- `sequential: true` on a tool acquires a per-run mutex (`Semaphore` from `lib/concurrency.ts`)
- Semaphore is a simple promise queue вҖ” no timeout, unbounded

---

## 7. Output Schema Handling (`lib/execution/output_schema.ts`)

- If agent has `outputSchema`, a `final_result` tool is injected automatically
- Model must call this tool to produce structured output
- Zod validation applied; on failure, model is nudged with validation error and retried

---

## 8. Observability (`lib/otel/`)

- OpenTelemetry tracing via `@opentelemetry/api`
- Spans created for runs, tool calls, model requests
- `TelemetrySettings` configures span names, function IDs, metadata
- Tracing is optional вҖ” no-op if no OTEL provider configured
